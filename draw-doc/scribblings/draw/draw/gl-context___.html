interface

gl-context<%> : interface?

gl-context<%>对象表示使用OpenGL绘制到特定dc<%>实例的上下文。要获取gl-context<%>，请调用目标绘图上下文的get-gl-context。

只有画布dc<%>和bitmap-dc%对象包含make-gl-bitmap中的位图，才支持OpenGL（始终在Windows和Mac OS上，有时在Unix上）。普通dc<%>绘图和OpenGL绘图可以混合在bitmap-dc%中，但使用'gl样式支持OpenGL的画布不可靠地支持普通dc<%>绘图；如果需要混合绘图模式，请使用位图，并使用画布最大化OpenGL性能。

当bitmap-dc%的目标位图通过set-bitmap更改时，关联的gl-context<%>将更改。画布上下文通常是双缓冲的，位图上下文是单缓冲的。

racket/gui/base库不提供OpenGL例程。相反，它们必须从单独的库中获得，如sgl。racket/gui/base中的工具只管理当前的OpenGL上下文，将其连接到Windows和位图。

在所有线程和事件空间中，一次只能有一个OpenGL上下文处于活动状态。OpenGL上下文不受线程间干扰的保护；也就是说，如果一个线程选择其OpenGL上下文之一，那么其他线程可以通过OpenGL命令写入上下文。但是，如果所有线程仅在传递给调用为当前线程的thunk内发出OpenGL命令，那么从单独的线程中进行绘制将不会产生干扰，因为调用为当前线程使用锁来序列化机架中所有线程的上下文选择。

method

(send a-gl-context call-as-current thunk    
  [alternate    
  enable-breaks?]) → any/c
  thunk : (-> any)
  alternate : evt? = never-evt
  enable-breaks? : any/c = #f

使用此OpenGL上下文作为OpenGL命令的当前上下文调用thunk。

该方法阻止获取保护全局OpenGL上下文的锁，并在thunk返回或逸出时释放该锁。锁是可重入的，因此在具有相同OpenGL上下文的同一线程中嵌套使用该方法不会获取或释放锁。

锁可以防止OpenGL使用线程之间的干扰。如果线程在保持上下文锁的同时终止，则释放该锁。继续跳到thunk中，不要获取锁或设置OpenGL上下文。有关干扰的详细信息，请参见gl-context<%>。

方法接受一个备用的可同步事件（synchronizable event），以便在为上下文锁阻塞时使用；另请参见sync。

方法调用的结果是thunk（如果调用）的结果，或者如果选择替代事件（而不是上下文锁）的结果。

在调用ok?方法时如果此方法返回#f，然后an exn:fail:contract exception is raised。

如果enable-breaks?为真，则该方法在为上下文设置lock而不是sync进行阻塞时使用sync/enable-break。

method

(send a-gl-context get-handle) → cpointer?

返回平台基础上下文的句柄。指针表示的值取决于平台：

  • Windows: HGLRC

  • Mac OS: NSOpenGLContext

  • Unix: GLXContext

请注意，这些值不是最“底层”上下文对象所必需的，但是对Racket有用。例如，NSOpenGLContext包装CGLContextObj。

method

(send a-gl-context ok?) → boolean?

如果此上下文可用，则返回 #t；否则返回#f。

如果在编译时或运行时禁用OpenGL支持，或者上下文与没有选定位图的bitmap-dc%关联，或者与单色选定位图关联，或者上下文用于不再存在的画布，或者在准备上下文时出现低级错误，则上下文不可用。

method

(send a-gl-context swap-buffers) → void?

为与画布关联的上下文交换前（可见）和后（OpenGL绘图）缓冲区，并且对位图上下文没有影响。

此方法隐式使用call-as-current获取上下文锁。但是，由于锁是可重入的，因此可以在调用中安全地使用swap-buffers方法作为call-as-current的thunk。
